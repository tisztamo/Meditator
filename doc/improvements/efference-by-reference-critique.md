# Critique: Efference by Reference

**Date:** 2026-06-30
**Subject:** Review of [efference-by-reference.md](efference-by-reference.md)

## Summary

The diagnosis in the design note is sound: exact facts can now reach the conscious
frame, but they still cannot reliably cross the action boundary. `m-act` gives the
realizer only a short stream tail and an intent gist, then asks it to generate tool
arguments under a small token budget. That means any exact, sizable payload must be
regenerated by value, which is exactly where the ARC checker fabricated grids.

The proposed direction is also right at the level of principle: action needs a way
to carry exact known content by reference. But the proposed mechanism, raw
`«handle»` expansion inside arbitrary string arguments, is too permissive and too
hard to make safe. It is a macro system, not a reference system.

## What is strong

- The problem is at the efferent boundary, not just in memory or model quality.
- A handle menu in the realizer frame is the right fix for blindness without dumping
  large fact values into the prompt.
- The realizer should spend tokens on procedure and intent, while substrate code
  carries exact data losslessly.
- The fix belongs in components (`m-act`, `m-facts`, and eventually other knowing
  components), not in the ARC solver.

## Main objection

Do not expand handles "anywhere in a hand's string arguments."

For terminal execution, string expansion cannot reliably know whether a placeholder
is in a safe data position. A fact can contain quotes, heredoc delimiters, shell
syntax, Python delimiters, or content that changes the meaning of the surrounding
script. The current note recognizes this as an injection caveat, but the milestone
still proposes context-free expansion before `cap.execute`.

That is not a small implementation detail. It is the central safety property.

Example of the risk:

```python
data = """«puzzle»"""
```

If the resolved fact contains `"""`, the reference escapes the string. Shell heredocs
have the same problem with delimiter collisions. JSON strings have escaping rules.
URLs, email bodies, Markdown, and form fields all have different rules. A generic
text splice cannot enforce "data, not code" across all hands.

## Better model: reference attachments

Replace transclusion with typed, policy-checked reference passing.

Instead of:

```python
grids = parse("""«puzzle»""")
```

Have the realizer call a hand with explicit references:

```json
{
  "language": "python",
  "script": "import json\npuzzle = json.load(open('refs/puzzle.json'))\n...",
  "refs": [
    { "handle": "fact:puzzle", "as": "puzzle.json", "encoding": "utf8" }
  ]
}
```

Then `m-act` resolves references as data and hands them to the selected capability
as metadata. The capability decides how references enter its world.

For `m-terminal`, the first implementation should materialize references as
read-only files in the per-run workspace, under a stable directory such as `refs/`.
`m-terminal` already writes generated scripts to a run directory before execution,
so this matches the existing shape of the component.

For other hands:

- note/message hands may opt into inline quoted rendering with sink-specific
  escaping;
- web/API hands may use multipart bodies or JSON fields;
- risky hands may reject refs entirely by default.

This keeps the general component-level contract without pretending all sinks share
one safe textual insertion rule.

## Proposed contract

Knowing components should register reference resolvers, not macro expanders.

Resolver result:

```js
{
  handle: "fact:puzzle",
  mediaType: "text/plain",
  text: "...",          // or bytes later
  chars: 2048,
  sha256: "...",
  source: "fact",
  policy: {
    private: false,
    executable: false
  }
}
```

Capabilities should declare reference policy:

```js
references: {
  mode: "files",        // "files", "inline", or "forbidden"
  maxBytes: 200000,
  allowedSources: ["fact"]
}
```

`m-act` should enforce this before execution:

1. parse and schema-validate tool arguments;
2. validate any `refs` array against the selected capability's reference policy;
3. resolve exact handles through registered resolvers;
4. pass resolved refs to `cap.execute` in a separate field, not by mutating arbitrary
   string args;
5. journal handle metadata, not large payloads.

## Handle rules

Use namespaced, exact handles for efferent references.

`recall-fact` can keep exact-or-prefix lookup because it is an interactive recall
hand. Execution should not. Prefix matching at action time can silently bind the
wrong datum if two similar facts appear later.

Recommended handle shape:

- `fact:puzzle`
- `fact:verdict:crop-rule`
- later, possibly `memory:span:<id>` or `perception:<id>`

The realizer frame should advertise a compact menu:

```text
## Exact references available
- fact:puzzle: seeded, pinned, 1842 chars, text/plain
- fact:verdict:crop-rule: earned, 73 chars, text/plain
```

Do not include payloads in this menu.

## Audit and privacy

The current `acted` event records `args`. If macro expansion happens before
execution, this can dump huge or sensitive facts into backstage memory. If expansion
does not happen in the journal, the audit trail no longer shows what actually ran.

Reference attachments give a cleaner audit record:

```js
{
  capability: "terminal",
  args: {
    language: "python",
    script: "...",
    refs: [{ handle: "fact:puzzle", as: "puzzle.json" }]
  },
  resolvedRefs: [
    {
      handle: "fact:puzzle",
      mountedAs: "refs/puzzle.json",
      chars: 1842,
      sha256: "..."
    }
  ]
}
```

This is reconstructable without copying the whole fact into the action journal.

## Revised smallest milestone

The safer first milestone:

1. `m-facts` registers a `fact:` resolver over exact keys.
2. `m-facts` publishes a retained reference menu: key, source, pinned flag, char
   count, optional media type, and maybe a one-line gloss.
3. `m-act` includes that menu in `_realizeFrame`.
4. `m-terminal` schema gains an optional `refs` array.
5. `m-act` resolves refs after JSON/schema validation and before execution.
6. `m-terminal` materializes resolved refs as read-only files under `refs/` in the
   run directory.
7. The realizer prompt instructs scripts to read reference files, never retype facts.
8. `acted` records reference metadata: handle, mounted path, chars, sha256.

Validation should include hostile fact contents that would break raw string
expansion, such as embedded Python triple quotes, shell heredoc delimiters, and JSON
escape sequences. The test passes only if those bytes are read as inert data and
never execute.

## Bottom line

Keep the design's principle: exact known content must leave the mind by reference,
not regeneration.

Change the mechanism: use typed reference attachments with per-hand policy. Do not
use global textual macro expansion.
